# Hermes Windows 一键部署工具 — 实现计划书

## 一、项目概述

### 1.1 项目目标

开发一个 **Windows 一键安装引导器**，让普通 Windows 用户在 15 分钟内完成 Hermes Agent 的完整部署，无需接触任何命令行操作。

### 1.2 用户画像

- **主要用户：** 国内 Windows 10/11 用户，对 AI 感兴趣但不熟悉 Linux
- **技术水平：** 会用浏览器、会点"下一步"，不会/不愿使用终端
- **设备环境：** Windows 10 2004+ 或 Windows 11，8GB+ 内存，20GB+ 可用磁盘

### 1.3 核心价值

把「需要会 Linux、会终端、会配环境」才能用的 Hermes Agent，变成「下载 → 双击 → 等 15 分钟 → 浏览器自动打开就能用」。这是目前社区还没人做的「最后一公里」。

### 1.4 不做的事

- ❌ 不自己写 Web UI — 用社区成熟的 hermes-web-ui（3148 stars, MIT 协议）
- ❌ 不魔改 Hermes 核心 — 只做外围包装和自动化
- ❌ 不做 macOS/Linux 版 — P0 专注 Windows

## 二、技术选型

| 模块 | 技术选型 | 理由 |
|------|----------|------|
| 安装引导器 | PowerShell 5.1+ | Windows 原生自带，零依赖 |
| 打包工具 | Inno Setup 6 | 免费、成熟、中文支持好、可按需捆绑文件 |
| WSL 发行版 | Ubuntu 24.04 LTS | WSL 默认推荐，社区支持最好 |
| Hermes 安装 | 官方 install.sh | 不重复造轮子 |
| Web UI | hermes-web-ui (npm 全局安装) | 社区成熟项目，覆盖全部管理需求 |
| 开机自启 | systemd + Windows Task Scheduler | WSL2 原生支持 systemd |
| 网络加速 | 国内镜像源（清华/中科大/阿里） | 解决国内 GitHub/npm 慢的问题 |
| 安装 UI | WPF 或 WinForms（C#） | 比纯 PowerShell GUI 更好看、更可控 |
| 项目分发 | 单个 .exe 安装包 | 用户只需下载一个文件 |

## 三、分阶段实现计划

### Phase 1：MVP — 命令行版一键安装脚本（2-3 天）

**目标：** 一份 PowerShell 脚本，用户右键"以管理员身份运行"即可完成全部安装。

**交付物：**
- `install-hermes.ps1` — 全自动安装脚本
- `README.md` — 用户使用说明

**核心任务：**

| # | 任务 | 预估时间 | 依赖 |
|---|------|----------|------|
| 1.1 | 环境检测模块（Win版本、WSL状态、磁盘空间、内存） | 2h | — |
| 1.2 | WSL2 自动安装 + Ubuntu 部署 | 3h | 1.1 |
| 1.3 | 国内镜像源自动配置（apt/pip/npm/git） | 2h | 1.2 |
| 1.4 | Hermes 自动安装（install.sh + setup） | 2h | 1.3 |
| 1.5 | hermes-web-ui 自动安装配置 | 1.5h | 1.3 |
| 1.6 | systemd 服务创建 + 开机自启配置 | 2h | 1.4, 1.5 |
| 1.7 | 安装进度显示 + 错误处理 + 日志 | 3h | 全部 |
| 1.8 | 测试：Win10/Win11 各测一轮 | 4h | 全部 |

**关键设计决策：**
- 脚本需支持「断点续装」— 某步失败后修复环境再跑，跳过已完成步骤
- 必须处理 WSL 安装需要重启的情况（提示用户重启后自动继续）
- 所有网络操作必须有超时和重试机制

### Phase 2：GUI 安装向导 + 打包（3-4 天）

**目标：** 把 Phase 1 的脚本包装成带 GUI 的安装向导，打包为 .exe。

**交付物：**
- `HermesInstaller.exe` — 带 GUI 的安装向导（Inno Setup 打包）
- 视觉素材（安装界面、图标等）

**核心任务：**

| # | 任务 | 预估时间 | 依赖 |
|---|------|----------|------|
| 2.1 | 设计 GUI 安装向导流程（步骤图、文案） | 2h | Phase 1 |
| 2.2 | WPF/WinForms 前端开发（欢迎页→检测→安装→完成） | 6h | 2.1 |
| 2.3 | GUI 与 PowerShell 后端通信 | 3h | 2.2, Phase 1 |
| 2.4 | Inno Setup 打包脚本 | 3h | 2.3 |
| 2.5 | 应用图标 + 安装界面设计 | 2h | — |
| 2.6 | 全流程集成测试 | 4h | 全部 |

**关键设计决策：**
- GUI 用 C# WPF，可以嵌入 PowerShell 输出流实时显示日志
- 每个安装步骤有独立的进度条和状态指示
- 「高级模式」按钮允许手动修改配置（镜像源、端口等）
- 安装器本身不需要管理员权限，但执行 WSL 安装时会触发 UAC 提权

### Phase 3：桌面增强功能（5-7 天）

**目标：** 实现文档中规划的桌面端差异化功能。

**交付物：**
- Agent Chat 内嵌对话（hermes-web-ui 增加聊天面板）
- Windows 右键菜单集成
- HITL 审批面板

**核心任务：**

| # | 任务 | 预估时间 | 依赖 |
|---|------|----------|------|
| 3.1 | hermes-web-ui 添加 `/api/chat` SSE 端点 | 4h | Phase 2 |
| 3.2 | hermes-web-ui 前端 ChatPage 组件开发 | 6h | 3.1 |
| 3.3 | Windows 右键菜单注册表项 | 2h | Phase 2 |
| 3.4 | 右键菜单 HTTP API 桥接服务 | 3h | 3.3 |
| 3.5 | HITL 审批 WebSocket 推送 + 面板 UI | 6h | 3.1 |
| 3.6 | MCP/ACP Tool Hub 管理页面 | 6h | 3.1 |
| 3.7 | 集成测试 + 文档 | 4h | 全部 |

**关键设计决策：**
- Agent Chat 不从头写，复用 hermes gateway 已有的消息处理逻辑
- 右键菜单用纯注册表 + HTTP API，不需要安装额外二进制
- 审批面板用 WebSocket，前端弹出模态框，超时自动拒绝

## 四、Phase 1 详细实施步骤

### 4.1 install-hermes.ps1 脚本结构

```
install-hermes.ps1
├── [配置区] 用户可修改的变量（镜像源、端口等）
├── [工具函数]
│   ├── Write-Step     — 带颜色的步骤输出
│   ├── Write-Error    — 错误输出 + 写入日志
│   ├── Test-Command   — 检查命令是否可用
│   ├── Invoke-Retry   — 带重试的命令执行
│   └── Get-UserHome   — 获取当前用户目录（处理管理员运行的情况）
├── [Step 0] 环境检测
│   ├── Win版本 >= 10.0.19041
│   ├── 管理员权限检查
│   ├── 磁盘空间 >= 20GB
│   ├── 内存 >= 8GB
│   └── 虚拟化是否启用（BIOS VT-x/AMD-V）
├── [Step 1] 安装 WSL2
│   ├── 检测 WSL 是否已安装
│   ├── wsl --install --no-distribution（安装 WSL 内核）
│   └── 如需重启：提示 + 注册 RunOnce 自动继续
├── [Step 2] 安装 Ubuntu
│   ├── wsl --install -d Ubuntu-24.04
│   ├── 等待 Ubuntu 初始化完成
│   └── 创建用户（非 root）
├── [Step 3] 配置国内镜像源
│   ├── Ubuntu apt → 清华/阿里镜像
│   ├── pip → 清华镜像
│   ├── npm → 淘宝镜像
│   └── git clone → 通过 gitee 镜像或代理
├── [Step 4] 安装 Hermes
│   ├── apt install python3.11 curl git
│   ├── curl install.sh | bash
│   └── hermes setup（非交互式，预置配置）
├── [Step 5] 安装 hermes-web-ui
│   ├── 安装 Node.js 22（nodesource）
│   ├── npm install -g hermes-web-ui
│   └── 配置端口（默认 8648）
├── [Step 6] 配置开机自启
│   ├── 创建 /etc/systemd/system/hermes.service
│   ├── 创建 Windows Task Scheduler 任务
│   └── 验证：手动启动 → 浏览器打开 localhost:8648
├── [Step 7] 安装完成
│   ├── 输出安装摘要
│   ├── 自动打开浏览器
│   └── 提示桌面快捷方式位置
```

### 4.2 systemd 服务配置细节

```ini
[Unit]
Description=Hermes Agent Service
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=%USER%
WorkingDirectory=/home/%USER%/.hermes
ExecStart=/home/%USER%/.local/bin/hermes gateway
Restart=always
RestartSec=10
Environment=HOME=/home/%USER%
Environment=PATH=/home/%USER%/.local/bin:/usr/local/bin:/usr/bin:/bin

[Install]
WantedBy=default.target
```

注意点：
- `After=network-online.target` 而非 `network.target`，确保网络完全就绪
- `RestartSec=10` 给 WSL 网络栈初始化留时间
- 必须显式设置 HOME 和 PATH 环境变量

### 4.3 国内网络加速策略

```
1. apt 源：sed 替换 sources.list → mirrors.tuna.tsinghua.edu.cn
2. pip 源：pip config set global.index-url https://pypi.tuna.tsinghua.edu.cn/simple
3. npm 源：npm config set registry https://registry.npmmirror.com
4. git clone：优先用 Gitee 上已有的 Hermes 镜像，没有则设置 git 代理
   或通过 GHProxy：https://ghproxy.com/https://github.com/...
5. 备选方案：在脚本内预置关键文件离线包，网络完全不通时走离线安装
```

### 4.4 错误处理与断点续装

- 每个 Step 执行前后写入状态文件 `~/.hermes/install-state.json`
- 脚本启动时读取状态，跳过已成功的 Step
- 失败 Step 输出详细错误日志，用户可选择「重试」「跳过」「中止」
- 日志写入 `~/hermes-install.log`，方便排查

## 五、潜在困难与风险

### 5.1 WSL 兼容性问题

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| BIOS 虚拟化未开启 | 高 | 致命 | Step 0 检测并给出 BIOS 设置图文教程 |
| Windows 版本过旧（<19041） | 中 | 致命 | 检测并拦截，提示升级 Windows |
| WSL 安装后需要重启 | 高 | 体验差 | 提示用户重启，注册 RunOnce 自动继续 |
| 企业电脑组策略禁用 WSL | 低 | 致命 | 检测并告知 IT 管理员 |
| WSL 内核更新失败 | 中 | 高 | 提供手动下载 wsl_update_x64.msi 的备用方案 |

### 5.2 网络问题（国内特有问题）

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| GitHub 无法访问 | 高 | 高 | GHProxy 代理 / Gitee 镜像 |
| npm 安装超时 | 高 | 中 | 淘宝镜像 + 增加超时时间 |
| pip 安装慢 | 高 | 中 | 清华镜像 |
| raw.githubusercontent.com 不通 | 高 | 致命 | 脚本内预埋 install.sh 内容 / 自建 CDN |
| npm 全局安装权限问题 | 中 | 中 | 配置 npm prefix 到用户目录 |

### 5.3 权限与安全

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| 杀毒软件拦截 PowerShell 脚本 | 中 | 高 | 代码签名 / 文档说明 |
| Windows 防火墙拦截 8648 端口 | 中 | 中 | 安装脚本自动添加防火墙规则 |
| 用户以管理员身份运行导致路径错乱 | 高 | 中 | 脚本内正确处理 HOME 目录 |
| UAC 弹窗过多 | 高 | 体验差 | 尽量减少提权次数，wsl --install 一次搞定 |

### 5.4 持续运行问题

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| WSL 空闲后自动关闭 | 高 | 中 | systemd 服务保持 WSL 活跃 |
| Windows 重启后 WSL 未自动启动 | 中 | 高 | Task Scheduler + 启动脚本双重保障 |
| hermes-web-ui 端口冲突 | 低 | 低 | 安装时检测端口，可选自定义 |
| WSL 内存泄漏导致系统卡顿 | 中 | 中 | 配置 .wslconfig 限制内存（建议 4GB） |

### 5.5 长期维护风险

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| hermes-web-ui 上游断更 | 高 | Fork 维护 / 提前评估替代方案 |
| Hermes install.sh 变化 | 中 | 版本锁定 / 定期 CI 测试 |
| WSL 内核更新破坏兼容性 | 低 | 延迟更新 / CI 矩阵测试 |

## 八、技术债务与后续规划

### 8.1 已知技术债务

1. WSL 不是真正的原生 Windows 应用，内存开销大（建议后续支持 Docker 方案）
2. PowerShell 脚本在非英文 Windows 上可能编码问题
3. hermes-web-ui 的 Chat 功能是社区贡献，需跟进上游合并

### 8.2 v2.0 展望

- Direct Windows 原生支持（如果 Hermes 后续支持 Windows 原生运行）
- 自动更新机制（检测新版本 → 一键升级）
- 多 Agent 协作面板
- 桌面悬浮球快捷入口
